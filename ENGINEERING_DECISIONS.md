# WardReady — Engineering Decisions

## Architecture

### Decision: Riverpod + GoRouter + Drift Stack
**Why:** Clean separation of concerns with reactive state management, declarative routing, and type-safe database access.
**Evidence:** 
- `pubspec.yaml` explicitly locks these packages
- Consistent provider patterns across all features
- GoRouter with redirect functions in `main.dart`
- Drift `AppDatabase` with generated `.g.dart` file
**Alternatives rejected:** setState, provider, bloc, hive, shared_preferences for main storage, Navigator 1.0
**Tradeoffs:** Learning curve for junior devs; boilerplate for simple state; strong typing reduces runtime errors
**Confidence:** HIGH — Stack is consistent and complete

### Decision: Offline-first with Supabase Sync
**Why:** Target users in Ethiopian medical schools have unreliable connectivity; educational content must be available offline.
**Evidence:** 
- `article_repository.dart:121-150` returns local cache on network errors
- `connectivity_notifier.dart` actively monitors network state
- `offline_banner.dart` shows when offline
- `database_recovery_screen.dart` handles corruption
**Alternatives rejected:** Online-only, cached_network_image only, connectivity-aware lazy loading
**Tradeoffs:** Storage pressure, sync complexity, data staleness
**Long-term consequences:** FTS5 index maintenance, backup/restore complexity, large APK size
**Confidence:** HIGH — Core pattern repeated throughout

## Repository Pattern

### Decision: Repository owns sync + cache consistency
**Why:** Single source of truth for remote + local data with automatic fallback.
**Evidence:** 
- `article_repository.dart` wraps all Supabase operations with local cache fallback
- `quiz_repository.dart` does same for questions
- Repositories receive callbacks to notify `connectivityProvider` and `syncStateProvider`
**Alternatives rejected:** Separate remote + local repositories, repository with explicit cache invalidation
**Tradeoffs:** Repositories become large; tight coupling to state providers
**Confidence:** HIGH

## Riverpod Usage

### Decision: AsyncNotifierFamily for feature state, StateProvider for simple state

## Supabase

### Decision: SupabaseClient singleton with RLS-aware error handling
**Why:** Auth + Postgres + Storage in one SDK; RLS provides backend security.
**Evidence:** 
- `Supabase.instance.client` used directly in providers

## Architectural Turning Points

### 1. Adoption of Offline-First (Inferred)
The entire repository structure suggests offline-first was foundational. Articles, quizzes, and progress all work without network. Evidence: multiple repository methods returning local cache on any network error.

### 2. Introduction of Theme System
Theme is in `core/theme/app_theme.dart` with semantic colors. All screens use `Theme.of(context).colorScheme.*`. This appears to be early decision — no legacy color constants found.

### 3. Repository Abstraction (Inferred)
All data access goes through repositories. `ArticleRepository` and `QuizRepository` follow identical patterns. This suggests repositories were decided early and templated.

### 4. SM-2 via Repository Integration
Spaced repetition is in a separate service but integrated into repository flow. The `quiz_table` has `next_due_at`, `ease_factor`, `repetitions` columns. Decision to extend DB schema for SM-2.

### 5. GoRouter Redirect Pattern
Admin guard and subscription guard both use GoRouter redirects. This suggests deliberate pattern adoption rather than screen-level checks.

## Design Principles

### Consistently Followed
- **Offline-first:** All repositories return local cache on error
- **Riverpod ownership:** All state in providers, no scattered state
- **Feature isolation:** Each feature in its own folder
- **Graceful degradation:** Offline banner, sync fallbacks everywhere

### Violated
- **Repository ownership:** QuizNotifier touches three services (violates single source of truth)
- **Provider ownership:** `allArticlesProvider` rebuilds on ANY article change (over-broad)
- **Minimal dependencies:** `fsrs` and `google_fonts` in pubspec but unused (dead weight)

## Most Elegant Parts

### 1. SpacedRepetitionService SM-2 Calculation
The `_calculateSchedule` method (spaced_repetition_service.dart:182-211) is clean, mathematical, and isolated. SM-2 math is not mixed with DB or UI code. Easy to verify against algorithm spec.

### 2. GoRouter Redirect Guards
Admin and subscription guards in `main.dart` (151-159) are declarative and centralized. No UI needs to check `isAdmin` — router handles it.

### 3. Connectivity Notifier Separation
Two providers (`connectivityProvider` and `serverUnreachableProvider`) handle device network vs server status separately. Smart distinction.

### 4. Category Progress Provider Family
`categoryProgressProvider` (category_progress_provider.dart) is a family provider keyed by category. Efficient caching per-category.

---

## Most Fragile Parts

### 1. QuizNotifier Coupling
`QuizNotifier` depends on three services simultaneously. If any fails, quiz breaks. Testing requires mocking all three. Changes to one service may break others.

### 2. Sync Callback Chain
`ArticleRepository` constructor has 6 callbacks. Wiring in `article_repository.dart:234-251` is hard to trace. If callback order wrong, silent failures.

### 3. Study Sessions Table Dual Readers
Both `StreakNotifier` and `ProgressNotifier` query `study_sessions` independently. If one writes and one reads concurrently, inconsistent data possible.

### 4. FTS5 Index Rebuild Loop
`_ensureSearchIndex` loops over ALL articles for FTS5 indexing. With 250k articles, this blocks.

## Future Architectural Risks

### At 5,000+ Articles
- FTS5 rebuild still manageable
- Quiz questions selection in loop may slow

### At 50,000+ Articles  
- Article list pagination becomes critical (already implemented)
- Category progress queries fire 25+ at once on sync

### At 250,000 Questions
- `_selectWeighted200Questions` fires 17 queries in sequence
- Due card query scans entire table without index
- Memory pressure from 200-question exam load

### At 100,000 Users
- Admin dashboard fetches all users at once
- No pagination in admin user list
- Subscription check is per-user query

### Scaling Well
- Riverpod providers handle loading states cleanly
- Drift queries use parameter binding (SQL injection safe)
- Sync returns cache on failure (graceful degradation)
