import google.generativeai as genai
import json, time

API_KEY = "PASTE_YOUR_GEMINI_KEY_HERE"
OUTPUT_FILE = "wardready_articles.sql"
DELAY_SECONDS = 4

genai.configure(api_key=API_KEY)
model = genai.GenerativeModel("gemini-1.5-flash")

DISEASES = [
  ("Hypertension", "Cardiology"),
  ("Heart Failure", "Cardiology"),
  # add all diseases here
]

PROMPT_TEMPLATE = """
You are a medical content writer for WardReady —
an offline app for Ethiopian health science students.
Write a complete article for: {disease}
Output ONLY valid JSON. No markdown. No code fences.
Exact structure:
{{
  "definition": "1-2 sentence clinical definition",
  "epidemiology": "global + Ethiopia/SSA burden",
  "etiology": "causative agents by type",
  "pathophysiology": "mechanism 3-5 sentences",
  "clinicalFeatures": "symptoms and signs",
  "diagnosis": "clinical+labs+imaging; note if unavailable in Ethiopia",
  "treatment": "Ethiopian MoH/EFDA first-line FIRST, then alternatives",
  "complications": "key complications + timeframe",
  "ethiopianContext": "Ethiopia burden, MoH drugs, local patterns",
  "mnemonics": "key mnemonic if exists, else null"
}}
"""

def clean_json(text):
  text = text.strip()
  if "```" in text:
    lines = text.split("\n")
    text = "\n".join(l for l in lines if not l.strip().startswith("```"))
  return text.strip()

def generate_article(disease, category):
  prompt = PROMPT_TEMPLATE.format(disease=disease)
  try:
    response = model.generate_content(prompt)
    raw = clean_json(response.text)
    data = json.loads(raw)
    title = disease.replace("'", "''")
    cat = category.replace("'", "''")
    json_str = json.dumps(data).replace("'", "''")
    sql = f"""INSERT INTO articles (id, title, category, content, updated_at)
VALUES (uuid_generate_v4(), '{title}', '{cat}', $${json_str}$$::jsonb, now());\n"""
    return sql, True
  except Exception as e:
    print(f"  FAILED: {disease} — {e}")
    return None, False

sql_output, success_count, failed = [], 0, []
for i, (disease, category) in enumerate(DISEASES, 1):
  print(f"[{i}/{len(DISEASES)}] {disease}...")
  sql, ok = generate_article(disease, category)
  if ok:
    sql_output.append(sql)
    success_count += 1
  else:
    failed.append(disease)
  time.sleep(DELAY_SECONDS)

with open(OUTPUT_FILE, "w", encoding="utf-8") as f:
  f.write(f"-- WardReady Articles\n-- Total: {success_count}\n\n")
  for sql in sql_output: f.write(sql + "\n")

print(f"COMPLETE: {success_count}/{len(DISEASES)}")
if failed: print(f"Failed: {failed}")

SET status = 'active',
    expiry_date = now() + interval '365 days',

FROM subscriptions s
JOIN auth.users u ON s.user_id = u.id
ORDER BY s.activated_at DESC;

-- Article count by category:
SELECT category, COUNT(*) as count FROM articles
GROUP BY category ORDER BY count DESC;

SECTION 20: COLAB + GROQ SETUP — STEPS & PITFALLS
What We Built

A Google Colab script that calls Groq (free, no credit card) to generate 75 WardReady articles per night as ready-to-paste Supabase SQL, using an expanded 14-field article structure.
Steps Taken

Step 1 — Upgrade article structure from 10 to 14 fields New fields added: redFlags, approach, contraindications, clinicalPearls These make articles clinically richer and help students impress attendings.

Step 2 — Choose Groq over Google Gemini Groq is free (no card), fast, and llama-3.3-70b-versatile produces high-quality medical content. Get key at console.groq.com.

Step 3 — Install in Cell 1

!pip install groq

Step 4 — Build Cell 2 with these specific design choices:

    DISEASES as a list of tuples — easy to extend per night
    FIELD_DESCRIPTIONS as a dict — avoids long strings that break on copy-paste
    .replace() instead of .format() for prompt injection — critical (see pitfalls)
    String concatenation instead of f-strings for SQL — avoids indentation issues
    clean_json() strips code fences and finds the first { to last } — handles model preamble
    escape_for_sql() escapes any $$ inside content — prevents SQL injection into the delimiter
    14-field validation check — skips and logs any article missing fields

Step 5 — Cell 3 downloads the file

from google.colab import files
files.download("wardready_articles.sql")

Pitfalls Encountered (in order)

Pitfall 1 — Wrong Flutter version in Jules setup 3.44.0 and 3.24.3 both bundle Dart SDK below ^3.12.1. ✅ Fix: use flutter_linux_3.44.2-stable.tar.xz

Pitfall 2 — IndentationError: unexpected indent Chat rendering mangles indentation inside try/except blocks when copy-pasted. ✅ Fix: flatten function structure, avoid nested try/except alignment.

Pitfall 3 — SyntaxError: unterminated string literal Long prompt strings typed across multiple lines break when chat wraps them silently. ✅ Fix: store each field description as a separate dict entry on its own line, then build the prompt block programmatically.

Pitfall 4 — NameError: API key treated as variable

GROQ_API_KEY = gsk_abc123   # ❌ no quotes
GROQ_API_KEY = "gsk_abc123" # ✅ correct

Always wrap the key in double quotes.

Pitfall 5 — SyntaxError: Groq call missing parentheses

client = Groq "api_key=..."   # ❌
client = Groq(api_key="...")  # ✅

Pitfall 6 — KeyError on field names during .format() The JSON structure inside PROMPT_TEMPLATE contains { and } characters. Python's .format() treats every {word} as a placeholder and throws a KeyError on the first field name it hits. ✅ Fix: use .replace() instead:

# ❌ breaks
prompt = PROMPT_TEMPLATE.format(disease=disease, category=category)

# ✅ works
prompt = PROMPT_TEMPLATE.replace("{disease}", disease).replace("{category}", category)

This is the most important pitfall. Any future prompt that contains JSON structure must use .replace() not .format().
Rules for Future Colab Scripts

    Never use .format() when the prompt contains JSON with curly braces — always .replace()
    Never put long strings on one line — use a dict or list and join programmatically
    Always wrap API keys in quotes
    Always use Groq(api_key="...") with parentheses
    Cell 1 installs, Cell 2 runs, Cell 3 downloads — never merge them
    Paste SQL into Supabase in batches of 10-15, not all at once
    Check article count after each batch: SELECT COUNT(*) FROM articles;

pitfall 7: Groq 429 rate limit hits around article 55 with DELAY=2. Fix: set DELAY=8 and resume with remaining diseases in a new run.

BUG FIX 1 — generate_article returned 2 values instead of 3

CAUSE:

return data, name, True     ← on success (was: return data, True)
return None, None, False    ← on total failure (was: return None, False)
