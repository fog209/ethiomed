# WardReady — Release Readiness Audit

## Critical Blocking Issues

### 1. Notification Permission Flow (Android 13+)
- **Severity:** HIGH
- **File:** `lib/core/services/notification_service.dart`
- **Evidence:** Uses `flutter_local_notifications` with `requestNotificationsPermission()` at line 114, but no runtime permission request flow for Android 13+ (POST_NOTIFICATIONS is declared but no code to request permission)
- **Risk:** App crashes or fails to show notifications on Android 13+
- **Fix:** Add explicit permission request in SettingsScreen before enabling reminders
- **Effort:** 2h

### 2. key.properties Required for Release
- **Severity:** HIGH (build blocker)
- **File:** `android/app/build.gradle.kts:36-42`
- **Evidence:** Release signing config requires `keystoreProperties["keyAlias"]` etc. but no fallback for debug builds
- **Risk:** Release builds fail without key.properties file
- **Fix:** Create key.properties template or use `signingConfig signingConfigs.getByName("debug")` for dev
- **Effort:** 30m

### 3. Supabase Keys via Build Args
- **Severity:** HIGH (security)
- **File:** `lib/app/env.dart`
- **Evidence:** `Env` class uses `String.fromEnvironment` with empty defaults — keys MUST be passed at build time but empty defaults cause silent failures
- **Risk:** App builds but cannot connect to Supabase
- **Fix:** Add validation in main.dart that throws if keys are empty in release mode
- **Effort:** 30m

## False Positives

### "Network without null check"
- Repositories that appear to call Supabase without guard actually handle errors gracefully
- `SocketException` caught and falls through to cache return
- This is intentional offline-first design, not a bug

- This is intentional to show bookmark state changes in article lists
- Not a performance issue at current scale
